# winc.cpp

Windows-native port of [d4rks1d33/claude.cpp](https://github.com/d4rks1d33/claude.cpp).

Run **Claude Code** and **OpenCode** against a fully local LLM on Windows — no API costs, no data leaving your machine. One PowerShell installer, one PowerShell launcher.

The original repo targets Linux + macOS (with WSL2 for Windows). This port runs on bare Windows 10/11 using MSVC + CUDA, and ships a curated model menu sized for **16 GB NVIDIA GPUs** (RTX 4070 Ti, 4080, 5070 Ti, etc.).

---

## What is this?

`winc.cpp` wires three things together automatically:

| Component | Role |
|-----------|------|
| [llama.cpp](https://github.com/ggerganov/llama.cpp) | Runs the local GGUF model (CUDA or CPU) |
| [LiteLLM proxy](https://github.com/BerriAI/litellm) | Translates Anthropic API calls -> OpenAI-compatible llama.cpp endpoint |
| `launcher.ps1` | Generated by the installer, baked with your hardware's flags |

Claude Code thinks it's talking to Anthropic. It's actually talking to your GPU.

## Requirements

| | |
|---|---|
| OS | Windows 10 or 11, x64 |
| GPU | NVIDIA with 16 GB VRAM recommended (CPU also works, slowly) |
| RAM | 16 GB minimum, 32 GB recommended |
| CUDA | CUDA Toolkit 12.x — [download](https://developer.nvidia.com/cuda-downloads) |
| Compiler | Visual Studio 2022 Build Tools with the **VC C++ Workload** |
| Other | Git, CMake, Python **3.9–3.13** — all on `PATH` |

> **Python note:** use 3.9–3.13. Python 3.14 is too new — `litellm`'s native dependencies (e.g. `orjson`) have no prebuilt wheels for it yet and fall back to a Rust source build that fails. If your default `python` is 3.14, the installer will offer to grab Python 3.12 via winget and build the venv with that; your system `python` is left untouched.

The installer will offer to `winget install` anything missing — you don't have to pre-install the toolchain by hand. CUDA, MSVC, git, cmake, python all get prompted for. If you'd rather do it yourself or are without `winget`, the manual one-liners are:

```powershell
winget install Git.Git
winget install Kitware.CMake
winget install Python.Python.3.12
winget install Microsoft.VisualStudio.2022.BuildTools --override "--quiet --add Microsoft.VisualStudio.Workload.VCTools --includeRecommended"
winget install Nvidia.CUDA
```

Close and reopen PowerShell after manual installs so the new `PATH` is picked up.

## Quick start

**Easy mode (recommended):** double-click `install.cmd`, answer prompts. When it finishes, double-click `launch.cmd`.

`install.cmd` / `launch.cmd` are thin batch wrappers that:
- `cd` to this folder (so right-click-Run-with-PowerShell brokenness doesn't matter)
- pass `-ExecutionPolicy Bypass -NoProfile` to PowerShell
- keep the window open at the end with a "Press any key" so you can read errors

**Manual mode:** open a PowerShell here and run

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\install.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File .\launcher.ps1
```

Useful installer flags:

| Flag | Effect |
|------|--------|
| `-YesToAll` | Answer "yes" to every winget install prompt (unattended) |
| `-NoAutoDeps` | Skip winget — just warn about missing tools |
| `-NoBuild` | Skip building llama.cpp (reuse an existing build) |
| `-NoModels` | Skip the model download step |

The installer will:

1. Detect your GPU and VRAM via `nvidia-smi`
2. Check git, cmake, python, MSVC, nvcc — offer to `winget install` anything missing
3. Source `vcvars64.bat` from your VS Build Tools install so `cl.exe` is on PATH (no need for "Developer PowerShell for VS 2022")
4. Create a Python venv under `.\venv\` (using a 3.9–3.13 interpreter — installing Python 3.12 via winget if your default is too new) and install `litellm` + `huggingface_hub`
5. Clone llama.cpp into `.\llama.cpp\` and build it with MSVC (`-DGGML_CUDA=ON` if CUDA is present)
6. Probe `llama-server --help` to learn which optimization flags this build supports
7. Write an **isolated** `.\.claude-local\settings.json` (its own config dir, leaving your global `~/.claude` untouched) that disables the attribution header — fixing a ~90% KV-cache hit-rate penalty
8. Ask for an optional HuggingFace token (needed for gated models)
9. Detect your VRAM tier (6-8 / 16 / 24 GB+), default to it (you confirm), then show that tier's models and download what you pick
10. Generate `.\launcher.ps1`, baked with your hardware's flags
11. Add this folder to your user `PATH` so `winc` works from any new terminal

Every step shows an overall `[####----] step N/12` bar, and the slow ones (git clone, the llama.cpp build, pip, model download) draw a **live progress bar with elapsed time** so nothing looks frozen on a slower CPU. The build/pip/clone times are remembered in `.winc-timings.json`, so on a re-run those bars show a real **ETA** ("~3m12s left") instead of just elapsed time.

The installer is re-runnable: existing models and the existing llama.cpp build are kept; you can re-run it to add more models from the menu.

## Idempotency & moving between machines

**Re-running `install.cmd` is idempotent.** Each step checks before acting: deps are only `winget`-installed if missing, the venv is reused if `litellm` already imports (rebuilt only if its Python is incompatible), the llama.cpp build is reused unless you pass `-Rebuild`, models are skipped if already in `.\models\`, and `launcher.ps1` is regenerated deterministically. So you can run it as many times as you like.

**What's portable vs. what isn't.** Only the *source* files are committed (see `.gitignore`); the heavy artifacts are deliberately **not**, because they're machine- and path-specific:

| Item | Portable? | Why |
|------|-----------|-----|
| `install.cmd`, `install.ps1`, `winc.*`, `catalog.ps1`, `hf_get.py`, `litellm_run.py`, `README.md` | ✅ yes | plain source — this is what git tracks |
| `models/*.gguf` | ✅ yes (just copy) | plain data; copy them to skip re-downloading |
| `venv/` | ❌ no | a Python venv hardcodes absolute paths; not relocatable |
| `llama.cpp/build/` | ❌ no | binaries compiled for *this* machine's CUDA / CPU |
| `launcher.ps1`, `.claude-local/` | ❌ no | generated per-machine (regenerated on install) |

**To set up on another machine:** copy or `git clone` the repo (optionally copy `models/` across to avoid re-downloading), then run `install.cmd`. It rebuilds the venv and llama.cpp correctly for that hardware and regenerates the launcher. Don't copy `venv/` or `llama.cpp/` over — let the installer make them.

**Moving the folder on the same machine** (rename or relocate) is now safe: `launcher.ps1` and `winc` locate everything relative to their own folder (no baked absolute paths), and the venv's `python.exe` keeps working after a move. The only thing to update is the PATH entry if you [added one](#making-winc-available-everywhere) — point it at the new location.

## The `winc` command

After installing, a small CLI (`winc`) handles day-to-day model management and launching — without re-running the full installer.

```powershell
winc ls                      # list downloaded + available models
winc -d qwen3.6-27b          # download a model (by alias)
winc -s claude qwen3.6-27b   # start Claude Code on that model (sandboxed instance)
```

| Command | What it does |
|---------|--------------|
| `winc ls` | Lists models already in `.\models\`, then the full catalogue with aliases, sizes, and an `[installed]` marker |
| `winc -d <alias>` | Downloads a catalogue model (e.g. `winc -d gpt-oss-20b`) |
| `winc -d <repo> <file>` | Downloads any GGUF from HuggingFace (e.g. `winc -d bartowski/Some-Model-GGUF Some-Model-Q4_K_M.gguf`) |
| `winc -r <model>` | Deletes a downloaded model from `.\models\` (prompts first; add `-y` to skip) |
| `winc -s claude <model>` | Starts the local stack and launches **Claude Code** on `<model>` |
| `winc -s opencode <model>` | Same, but launches **OpenCode** |
| `winc -s cli <model>` | Same, but the raw llama.cpp chat CLI |
| `winc help` | Usage |

`<model>` is a catalogue alias (see `winc ls`) or any part of a downloaded filename, so `winc -s claude qwen3.6-27b` and `winc -s claude Qwen3.6-27B-Q3_K_M.gguf` are equivalent.

**`winc -s` reuses the exact isolated, hardware-tuned setup** described in [Running alongside your normal (Opus) Claude Code](#running-alongside-your-normal-opus-claude-code): it starts `llama-server` + LiteLLM, points the instance at the local proxy, and sets `CLAUDE_CONFIG_DIR` to `.\.claude-local\` — so it never touches your logged-in cloud Claude Code in another terminal. The only difference from `launch.cmd` is that the model and app are chosen for you instead of prompted.

### Making `winc` available everywhere

**The installer already does this** — its final step adds this folder to your user `PATH` (idempotent). Just open a **new** terminal after install and `winc ls` works from any directory.

From inside the folder it also works without PATH (`winc ls` in `cmd`, or `.\winc ls` in PowerShell). If you ever need to add it by hand:

```powershell
[Environment]::SetEnvironmentVariable('PATH', $env:PATH + ';C:\Claude\winc.cpp', 'User')
```

## Speculative decoding

Up to 2.5x faster generation by using a tiny 0.5B draft model to predict tokens in parallel, then verifying with the main model (no quality loss). Only offered when:

- Your llama.cpp build supports `--spec-draft-model`
- You picked a Qwen2.5-Coder model (the only family with a matched 0.5B draft on bartowski)
- You have **24 GB+ VRAM** — on 16 GB the draft model + extra KV cache leaves no headroom

If you have 24 GB+ and want it, say `y` when the installer asks.

## Launcher

Running `launcher.ps1` gives you:

```
========================================================
  AI LOCAL LAUNCHER (Windows)
  GPU: NVIDIA GeForce RTX 4070 Ti | Cores: 16 | Threads: 12
  RAM: 32 GB
========================================================

Available models:
  [0] Qwen3.6-27B-Q3_K_M.gguf                   (13.6 GB)
  [1] gpt-oss-20b-mxfp4.gguf                    (12.1 GB)

Select model [0]:

  [1] llama.cpp CLI     (direct chat)
  [2] Claude Code       (via LiteLLM proxy)
  [3] OpenCode          (via LiteLLM proxy)

Mode [2]:
```

It starts `llama-server.exe` on port 8080, starts `litellm` on port 4000, sets `ANTHROPIC_BASE_URL` to the proxy, then launches Claude Code or OpenCode. Ctrl+C in the launcher window cleans up both child processes.

## Hardware optimization details

The launcher is generated with flags tuned for your machine at install time:

| Flag | Effect | Condition |
|------|--------|-----------|
| `-ngl 99` | All layers on GPU | When CUDA was detected |
| `--cache-type-k q8_0` / `--cache-type-v q8_0` | Half-size KV cache | Always |
| `--flash-attn` | Faster attention, less VRAM | If `llama-server` supports it |
| `--mlock` | Pin model in RAM | Only when system RAM >= 32 GB |
| `--prio 2` | Lower scheduling jitter | If supported |
| `-b 2048 -ub 2048` | 2-3x faster prompt processing | GPU mode |
| `-DGGML_CUDA=ON` | CUDA at build time | nvcc on PATH |

Context window is auto-selected per VRAM at every launch:

| VRAM | Context |
|------|---------|
| 6 GB | 40,960 |
| 8 GB | 49,152 |
| 16 GB | **65,536 (default for this build)** |
| 24 GB+ | 131,072 |

## How the proxy works

```
Claude Code / OpenCode
        |  POST /v1/messages (Anthropic format)
        v
   LiteLLM proxy  :4000
        |  POST /v1/chat/completions (OpenAI format)
        v
   llama-server   :8080
        |
        v
   Local GGUF model (your GPU)
```

Claude Code sees `ANTHROPIC_BASE_URL=http://127.0.0.1:4000` and a dummy API key. LiteLLM rewrites the request format and forwards to llama.cpp. All traffic stays on localhost.

## Running alongside your normal (Opus) Claude Code

You can keep using your logged-in, cloud Claude Code in one terminal while running the **local** model in another — they're fully isolated:

- **Per-terminal env.** The launcher sets `ANTHROPIC_BASE_URL` / `ANTHROPIC_API_KEY` only in its own PowerShell process. A Claude Code session in any other terminal never sees them and keeps talking to Anthropic.
- **Separate config dir.** The launcher sets `CLAUDE_CONFIG_DIR` to a project-local `.\.claude-local\`, so the local instance has its **own** settings, credentials, history and todos. It does **not** read or write your global `%USERPROFILE%\.claude`, so there's no shared-credential or `/logout` collision, and no onboarding state bleed.
- **Global config untouched.** The installer writes its tuning (including `CLAUDE_CODE_ATTRIBUTION_HEADER=0`, which avoids a ~90% llama.cpp prefix-cache penalty) into `.\.claude-local\settings.json` — never into your global `~/.claude`.

So the typical workflow is two terminals:

```
Terminal 1:  claude                      # your normal Opus, cloud, logged in
Terminal 2:  launch.cmd  ->  Claude Code  # local quantized model, sandboxed
```

> If an earlier build of this installer already modified your global `%USERPROFILE%\.claude\settings.json`, it left a `settings.json.bak` next to it — those changes were harmless (telemetry/attribution off, no API redirect), but you can restore the backup if you want a pristine global config.

## Troubleshooting

**Claude Code shows the model as "Sonnet 4.6" — am I really on the local model?**

Yes (if you launched via `winc -s` / `launch.cmd`). Claude Code only accepts real Claude model names, so the LiteLLM proxy aliases your local GGUF as `claude-sonnet-4-6` and the launcher starts `claude --model claude-sonnet-4-6`. The "Sonnet 4.6" label is cosmetic — behind it your chosen model is running on your GPU. "API Usage Billing" just means an API key is set (a dummy one); all traffic stays on `127.0.0.1`, nothing is billed.

To **prove** it's local, while a prompt is generating:
```powershell
nvidia-smi                      # llama-server using ~13-14 GB VRAM + GPU load = local
Get-Content $env:TEMP\llama.log -Tail 5   # shows request/timing activity
```
If you instead launched `claude` directly (not through `winc`/`launch.cmd`), you're on your **normal cloud** Claude Code — there'll be no `winc.cpp` launcher banner above the welcome screen, and it really is the paid API. Always start the local one with `winc -s claude <model>`; you should see the `Ready / Proxy / Server / Config ...\.claude-local` banner first.

**`llama-server` not responding after launch**
```powershell
Get-Content $env:TEMP\llama.log -Tail 20
```
Common causes: out of VRAM (pick a smaller model or lower context), CUDA toolkit missing at build time (llama.cpp will have built CPU-only — re-run `install.ps1` after installing CUDA).

**LiteLLM not responding**
```powershell
Get-Content $env:TEMP\litellm.log -Tail 20
```
Most often port 4000 is held by an old process — `Get-Process python | Where-Object { $_.CommandLine -match 'litellm' } | Stop-Process -Force`.

**Claude Code still shows "API Usage Billing"**

Make sure no `ANTHROPIC_API_KEY` is set in your user env:
```powershell
[Environment]::SetEnvironmentVariable('ANTHROPIC_API_KEY', $null, 'User')
```
Then open a new PowerShell and rerun `launcher.ps1`.

**Model download fails**

Some models (Llama, Gemma) are gated and need a HuggingFace token. Get one at [huggingface.co/settings/tokens](https://huggingface.co/settings/tokens) and re-run `install.ps1`.

**Build fails — `nvcc` not found**

Install the CUDA Toolkit from [developer.nvidia.com/cuda-downloads](https://developer.nvidia.com/cuda-downloads), close and reopen PowerShell, run `nvcc --version` to confirm it's on PATH, then re-run `install.ps1`. Until CUDA is installed, llama.cpp will build CPU-only.

**Build fails — `cl.exe` not found**

You're missing the MSVC C++ workload. Install it:
```powershell
winget install Microsoft.VisualStudio.2022.BuildTools --override "--quiet --add Microsoft.VisualStudio.Workload.VCTools --includeRecommended"
```
Then run `install.ps1` from a fresh **Developer PowerShell for VS 2022** window (so `cl.exe` is on PATH).

## Adding models manually

Easiest: `winc -d <repo> <file>` (downloads any GGUF, e.g. `winc -d unsloth/Qwen3.6-27B-GGUF Qwen3.6-27B-Q3_K_M.gguf`).

Or drop any `.gguf` straight into `.\models\` — the launcher and `winc ls` auto-detect everything in that directory.

To pull one with Python directly (this is exactly what `winc -d` does under the hood — it uses the venv's `python.exe` + the `huggingface_hub` library rather than the `hf.exe` console script, which avoids breakage if the folder is ever renamed):

```powershell
.\venv\Scripts\python.exe -c "from huggingface_hub import hf_hub_download; hf_hub_download(repo_id='unsloth/Qwen3.6-27B-GGUF', filename='Qwen3.6-27B-Q3_K_M.gguf', local_dir='.\models')"
```

> For gated models (some Llama/Gemma repos), set a token first: `$env:HF_TOKEN = 'hf_...'`.
